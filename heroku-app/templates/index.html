User Stories Quality Analyzer

Try your user story and see if it has the necessary quality for the next stage to becoming the next software innovation.




Background

Scrum Methodology being one of the most popular technique for agile software development presents great working strategies to fast innovate and develop software, but one of the weak points is the construction of quality user stories they could make efficient the software delivery, to start tackling this problem we investigated the composition of the scrum methodology and the user stories and the type of user stories to start a brainstorm and find solutions of how to evaluate the quality of user story and give a fast feedback to the user so it could be correct and to be the closest to perfect user stories

REVIEWS

Well-formed

Before it can be considered a user story, the core text of the requirement needs to include a role and the expected functionality: the means. like the next example: "I want to see an error when I cannot see recommendations after I upload an article" does not adhere to this syntax, as it has no role. It is likely that the user story writer has forgotten to include the role. The story can be fixed by adding the role: ‘‘As a Member, I want to see an error when I cannot see recommendations after I upload an article.’’

Atomic

A user story should concern only one feature. Although common in practice, merging multiple user stories into a larger, generic one diminishes the accuracy of effort estimation. for example the next user story:"As a User, I am able to click a particular location from the map and thereby perform a search of landmarks associated with that latitude longitude combination" The user story consists of two separate requirements: the act of clicking on a location and the display of associated landmarks. This user story should separate

• ‘‘As a User, I’m able to click a particular location from the map’’

• ‘‘As a User, I’m able to see landmarks associated with the latitude and longitude combination of a particular location.’’

Unambiguous

Ambiguity is intrinsic to natural language requirements; the requirements engineer writing user stories has to avoid it to the extent this is possible. Not only should a user story be internally unambiguous, but it should also be clear in relationship to all other user stories. The Taxonomy of Ambiguity Types is a comprehensive overview of the kinds of ambiguity that can be encountered in a systematic requirements specification. In next story "As a User, I am able to edit the content that I added to a person’s profile page", ‘‘content’’ is a superclass referring to audio, video, and textual media uploaded to the profile page as specified in three other, separate user stories in the real-world user story set. The requirements engineer should explicitly mention which media are editable; for example, the story can be modified as follows:‘‘As a User, I am able to edit video, photo and audio content that I added to a person’s profile page.’’

Full sentence

A user story should read like a full sentence, without typos or grammatical errors. For instance, the next user story: "Server configuration" is not expressed as a full sentence (in addition to not complying with syntactic quality). By reformulating the feature as a full sentence user story, it will automatically specify what exactly needs to be configured. For example, we can modified to ‘‘As an Administrator, I want to configure the server’s sudo-ers.’’
